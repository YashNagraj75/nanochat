Nanochat is a from-scratch GPT pretraining stack built for efficiency on Hopper-class GPUs. It incorporates a number of architectural refinements over a baseline transformer: Rotary Position Embeddings (RoPE), Value Embeddings, a Smear Gate, Backout Lambda, and a combined weight initialization strategy designed for bfloat16 stability.

This article documents the design decisions encountered during construction, with interactive visualizations for the less intuitive components. Each section answers a specific question that arose while building the model from scratch.

01

Rotary Position Embeddings (RoPE)

[§]
TL;DR Rotate pairs of Q/K dimensions by an angle ∝ position. Dot-products then depend only on relative distance — absolute positions cancel out automatically.

Instead of adding a positional vector, RoPE rotates each (x₁, x₂) pair in the head dimension by angle m·θ, where m is the token position and θ is a per-dimension frequency.

new_x₁ = x₁·cos(mθ) − x₂·sin(mθ)
new_x₂ = x₂·cos(mθ) + x₁·sin(mθ)
⟳ Interactive RoPE Visualizer — drag the sliders
m = 3
d = 0 · θ = 1.0000
10 tokens
Why relative positions fall out Q[m] · K[n] = (R_m · q) · (R_n · k) = q · (R_m^T R_n) · k = q · R_{m-n} · k. The absolute m and n cancel — only the gap (m−n) survives in the score.

Code — stacks first and second halves, then applies the 2D rotation in parallel across all pairs:

def apply_rotary_emb(x, cos, sin):  # x: (B, T, H, D)
    d = x.shape[-1] // 2
    x1, x2 = x[..., :d], x[..., d:]
    x_rot = torch.cat([-x2, x1], dim=-1)  # rotate_half
    return x * cos + x_rot * sin

Frequencies precomputed once (registered as a buffer, never updated by the optimizer):

freqs = 1.0 / (10000 ** (torch.arange(0, head_dim, 2) / head_dim))
t     = torch.arange(seq_len)
freqs = torch.outer(t, freqs)                        # (T, D//2)
cos   = torch.cos(freqs).repeat_interleave(2, dim=-1)  # (T, D)
sin   = torch.sin(freqs).repeat_interleave(2, dim=-1)
# reshaped → (1, T, 1, D) for broadcast over (B, T, H, D)
Frequency intuition Low d → θ ≈ 1.0 → fast rotation → short-range patterns.
High d → θ → 0 → slow rotation → long-range context.
Together they give the model a multi-scale positional clock.
02

head_dim = n_embd ÷ n_head (not ÷ n_kv_head)

[§]
TL;DR head_dim is the per-head vector width, always split over query heads. GQA keeps head_dim identical across Q, K, V — only the number of K/V heads shrinks.
n_embd = n_head × head_dim  →  head_dim = n_embd ÷ n_head  (e.g. 768÷12 = 64)
GQA layout — n_embd=768, n_head=12, n_kv_head=4, head_dim=64
Common mistake Using n_kv_head in the denominator (768÷4=192) breaks every projection shape and makes K/V heads incompatible with Q.
TensorShapeHeads
Q(B, T, 12, 64)n_head query heads
K(B, T, 4, 64)n_kv_head key heads — same head_dim
V(B, T, 4, 64)each KV head shared by 3 Q heads

Value Embeddings (VE)

[§]
TL;DR Every other layer, inject the raw token embedding back into V via a learned gate [0, 3]. Prevents token identity from drowning in deep residual rewrites.

In a deep transformer the residual stream gets overwritten so many times that the original token identity fades. VE anchors it back on alternating layers:

gate = 3 * sigmoid(W_gate(x[..., :12]))  # (B, T, n_kv_head) ∈ (0, 3)
v    = v + gate.unsqueeze(-1) * ve         # ve = raw embedding from wte
Gate mixer — drag to see how v_proj vs. ve blend
w=0.00 · gate=1.50 · 50% identity
Gate ceiling = 3, not 1 The 3× scale lets ve fully dominate v — not just blend but override. Model can choose "ignore context, trust raw identity."
Cheap gate: first 12 dims only Avoids a full 768→n_kv_head projection. 12 channels is enough signal to compute one scalar per head.
05

Smear Gate

[§]
TL;DR Mix the previous token's embedding into the current one — bigram-level info injected before attention even runs. Starts at zero, opts in during training. 
gate   = sigmoid(W_smear(x[..., :24]))    # (B, T, 1) — scalar per token
x_prev = F.pad(x[:, :-1], (0,0,1,0))     # shift right by 1 position
x      = x + gate * smear_lambda.exp() * x_prev
Why Linear(24→1) 24 input channels (cheap, no full context needed) → 1 scalar: "how much previous token to blend?" One number per token.
smear_lambda starts at 0 exp(0)=1, so at init the model gently opts in. Same pattern as backout_lambda — both start conservative and grow as training proceeds.
06

Backout Lambda

[§]
TL;DR Subtract a fraction of the mid-layer residual from the final residual. Strips low-level noise so the LM head sees a cleaner, more abstract signal. 
x_final = x_final - backout_lambda * x_mid_layer
Signal decomposition — backout removes low-frequency residue
Image processing analogy Identical to unsharp masking: subtract the blurry low-frequency layer (x_mid) to make high-frequency detail (semantics, next-token predictions) stand out.
Starts at 0.2 (not 0) Authors already know it helps, so no opt-in needed. Compare with smear_lambda which starts at 0 and must be learned.
x_mid = mid layer Mid-layer residual still carries mostly low-level features (token identity, local syntax) before high-level processing dominates.

Weight Initialization — Uniform vs Normal, and √3

[§]
TL;DR Use Uniform(−√3σ, √3σ) instead of Normal for projections. Same variance, but hard-capped — no outlier tails to destabilise bfloat16 early in training.
Uniform(−a, a)  has  std = a/√3  →  set a = √3·σ to match Normal(0,σ²)
Distribution comparison — why Uniform wins for bfloat16 projections
DistributionstdTailsUsed for
Normal(0, σ²)σUnbounded (3σ, 4σ outliers)wte, lm_head
Uniform(−√3σ, √3σ)σHard cap at ±√3σall projections
σ = n_embd ** -0.5             # target std
a = 3**0.5 * n_embd**-0.5      # Uniform bound = √3·σ
torch.nn.init.uniform_(w, -a, a)

# Output projections start at zero → each block is identity at init
attn.o_proj → zeros
mlp.c_proj  → zeros
bfloat16 danger Normal tails (3σ, 4σ values) produce large dot products in attention, causing overflow/NaN in bfloat16 at step 1. Uniform eliminates this risk entirely.

Residual Lambdas — 1.15 → 1.05 Schedule

[§]
TL;DR Scale the residual stream by λ > 1 before adding each block. Linearly decays 1.15 → 1.05 from early to late layers. Amplifies signal, prevents vanishing gradients in deep nets. 
x = resid_lambda[i] * x + block(x)
# resid_lambdas[i] = 1.15 − 0.10 × i / (n_layer − 1)
# Layer 0 → 1.15 · · · Layer N−1 → 1.05
Lambda schedule — change layer count to see the curve
24 layers
Why λ > 1.0 Amplifies the existing residual before adding the block. Keeps gradient magnitude healthy through 24+ layers.
Why it decays early→late Early layers transform raw embeddings (low information → need more boost). Late layers refine rich representations (less boost needed). The exact values 1.15/0.10 are empirically tuned.

Total Training FLOPs — 6 × params + attn_flops

[§]
TL;DR Every linear layer costs 6 FLOPs per parameter per token during training (2 forward + 2 backward dA + 2 backward dW). Attention matmuls are added separately because they have no parameters — they scale with T², not param count. 
num_flops_per_token = 6 * (nparams - nparams_exclude) + attn_flops

For any linear layer y = x @ W, training needs three passes of equal cost:

PassOperationFLOPs
Forwardy = x @ W2 × params
Backward (input grad)dL/dx = dL/dy @ Wᵀ2 × params
Backward (weight grad)dL/dW = xᵀ @ dL/dy2 × params
Total6 × params

124M params
25M excluded
Why 2 FLOPs per param per pass Each weight participates in exactly one multiply-add per output element. One multiply + one add = 2 FLOPs. Backward passes are transposed matmuls of the same shape — same count.
Why embeddings are excluded A token lookup in wte is an index operation — O(1), no multiply-add. Applying 6× to these params would overcount. The lm_head (weight-tied to wte) does do a matmul and is typically included.
ComponentFLOPs per token
Linear layers (all proj, MLP)6 × num_params
Attention matmuls (Q@Kᵀ, scores@V)12 × n_embd × T_eff (see §10)
Embedding lookup (wte)0 — excluded via nparams_exclude

Attention FLOP Count — 12 × n_embd × q × seq_len

[§]
TL;DR The 12 = 2 (multiply-add) × 2 (two matmuls: Q@KT and scores@V) × 3 (training forward + 2 backward passes). Summed over all heads gives 12 × n_embd × T_q × T_k.

Every attention layer has two matrix multiplications. Each costs FLOPs during training's forward and backward passes:

FactorValueWhy
Multiply-add per element× 2One multiply + one add to compute each output element
Matmuls in attention× 2Q@KT (scores) and scores@V (output) — same shape cost
Training passes× 3Forward + dA = dC@BT + dB = AT@dC
Total122 × 2 × 3
FLOPs = 12 × head_dim × T_q × T_k   (per head)
      = 12 × n_embd  × T_q × T_k   (all heads, since n_head × head_dim = n_embd)
FLOP breakdown — drag to see how sequence length and n_embd affect compute
T = 2048
n_embd = 768
n_layer = 12
Quadratic in sequence length T_q × T_k = T² (causal average: T×T/2). Doubling the context quadruples attention FLOPs — the core motivation for sliding window attention and FlashAttention's IO-awareness.
"Effective" seq len in causal attention Token t attends to positions 0..t only. Average keys seen = T/2, so effective_seq_len ≈ T/2 for a full causal sequence.

The backward pass breakdown for one matmul C = A @ B:

# Forward
C  = A @ B          # compute scores / output
# Backward
dA = dC @ B.T       # gradient w.r.t. Q (or scores)
dB = A.T @ dC       # gradient w.r.t. K (or V)
# → 3 passes × same shape = 3× forward cost

Smear Gate — How the cat Works

[§]

The smear gate mixes the previous token's embedding into each token before attention runs.

gate = smear_lambda * sigmoid(smear_gate(x[:, 1:, :24]))  # (B, T-1, 1)
x = torch.cat([x[:, :1], x[:, 1:] + gate * x[:, :-1]], dim=1)

With T = 4 tokens, here is what happens step by step:

Why cat instead of F.pad?

F.pad inserts a zero vector at position 0 and then gates it — blending a fake signal into token A. The cat approach keeps token 0 exactly unchanged. No spurious signal for the first token.

Why x[:, 1:] for the gate input?

The gate is computed from the current token's first 24 channels — it asks "how much of my left neighbour do I want?" Token 0 has no left neighbour so it is simply excluded from the gate computation.

Logit Softcapping

[§]

Squashes logits into (−15, +15) smoothly — no hard boundary, no dead gradients.

softcap = 15
logits = softcap * torch.tanh(logits / softcap)

tanh vs hard clamp

Hard clamp
logits.clamp(-15, 15)

Gradient is zero outside [−15, 15]. Model can't learn from large logits.
Softcap (tanh)
15 * tanh(logits / 15)

Gradient is always nonzero (just shrinks). Training stays continuous.

Why cap at all?

Early in training, random projections produce huge logits (e.g. 200, −300). Softmax of those collapses — one class gets probability ≈ 1, everything else ≈ 0. The model is confidently wrong before learning anything: loss spikes, exploding gradients, bfloat16 overflow. Softcapping prevents this from step 1.

The value 15 — empirically tuned. Tight enough to stabilize early training, loose enough not to constrain a trained model's confidence. Popularized by Gemma 2.
